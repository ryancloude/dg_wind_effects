# Score Round Wind Model Schema

## Purpose

`score_round_wind_model` is the production scoring step for the round-level one-stage CatBoost weather-impact model.

It consumes:
- the canonical round-level model-input dataset from `gold_wind_model_inputs`
- a specific trained model artifact from `train_round_wind_model`

It produces:
- round-level predicted scores under observed conditions
- round-level predicted scores under reference counterfactual conditions
- estimated wind, temperature, and total weather impacts in strokes
- deterministic scored round parquet outputs
- event-level scoring checkpoints in DynamoDB

This step does not train the model.
This step does not produce dashboard aggregates directly.
Its responsibility is row-level production scoring.

---

## Input Contracts

## 1. Scored dataset input

### Source location
The scoring step reads round model-input parquet rows from:

- `gold/pdga/wind_effects/model_inputs_round/event_year=<YYYY>/tourn_id=<ID>/model_inputs_round.parquet`

### Grain
One row per:
- `(tourn_id, round_number, player_key)`

### Required input columns
The scoring step expects the round model-input dataset to contain at least:

- `event_year`
- `tourn_id`
- `round_number`
- `player_key`
- `course_id`
- `layout_id`
- `division`
- `player_rating`
- `actual_round_strokes`
- `round_strokes_over_par`
- `hole_count`
- `round_total_hole_length`
- `round_total_par`
- `round_wind_speed_mps_mean`
- `round_wind_speed_mps_max`
- `round_wind_gust_mps_mean`
- `round_wind_gust_mps_max`
- `round_temp_c_mean`
- `round_precip_mm_sum`
- `round_precip_mm_mean`
- `round_pressure_hpa_mean`
- `round_humidity_pct_mean`
- `round_wind_speed_bucket`
- `round_wind_gust_bucket`

Additional fields may also be present and passed through.

## 2. Trained model artifact input

The scoring step requires a specific trained artifact selected by:

- `training_request_fingerprint`

### Model artifact bundle
The scoring step loads:

- `model.cbm`
- `training_manifest.json`
- `feature_columns.json`
- `categorical_feature_columns.json`

from the training artifact prefix associated with the selected training fingerprint.

### Artifact dependency
The scoring step assumes the model artifact was created by `train_round_wind_model` and is compatible with the canonical round model-input schema.

---

## Model Selection Contract

The scoring step is intentionally explicit.

### Required selector
- `--training-request-fingerprint <fingerprint>`

The scoring step does not automatically use:
- latest model
- most recent successful model
- inferred artifact

This is deliberate for reproducibility.

---

## Scoring Feature Contract

The scoring step uses the exact feature contract stored in the training artifact.

### Feature source
Features are loaded from:
- `feature_columns.json`
- `categorical_feature_columns.json`

### Scoring preparation rules
The scoring step:
- verifies required scoring columns exist
- verifies all required model feature columns exist
- coerces numeric model feature columns to numeric
- drops rows with nulls in numeric model feature columns
- casts categorical feature columns to strings
- fills missing categorical values with `__MISSING__`

If no rows remain after these checks, the event fails scoring.

This is a strict production contract, not a permissive fallback path.

## Reference Counterfactual Policy

The scoring step produces one observed prediction and three counterfactual prediction sets.

## Observed prediction
- `predicted_round_strokes`

This is the model prediction under the observed round conditions.

## Wind reference prediction
- `predicted_round_strokes_wind_reference`

This is generated by replacing wind-related fields with the reference wind condition while leaving non-wind weather fixed.

### Wind reference values
- `wind_speed_reference_mph = 2.0`
- `wind_gust_reference_mph = 3.0`

### Wind reference fields changed
- `round_wind_speed_mps_mean`
- `round_wind_speed_mps_max`
- `round_wind_gust_mps_mean`
- `round_wind_gust_mps_max`
- `round_wind_speed_bucket`
- `round_wind_gust_bucket`

## Temperature reference prediction
- `predicted_round_strokes_temperature_reference`

This is generated by replacing temperature with the project’s selected temperature reference while leaving the rest fixed.

### Temperature reference value
- `temperature_reference_c = 12.0`

### Temperature reference fields changed
- `round_temp_c_mean`

## Total weather reference prediction
- `predicted_round_strokes_total_weather_reference`

This is generated by replacing weather-related fields with the full reference-weather condition.

### Total weather reference values
- `wind_speed_reference_mph = 2.0`
- `wind_gust_reference_mph = 3.0`
- `temperature_reference_c = 12.0`
- `precip_reference_mm = 0.0`
- `pressure_reference_hpa = training median`
- `humidity_reference_pct = training median`

### Total weather reference fields changed
- wind fields
- wind bucket fields
- `round_temp_c_mean`
- `round_precip_mm_sum`
- `round_precip_mm_mean`
- `round_pressure_hpa_mean`
- `round_humidity_pct_mean`

## Estimated Impact Fields

The scoring step produces these model-based impact fields.

## Estimated wind impact
- `estimated_wind_impact_strokes`

Definition:
- `predicted_round_strokes - predicted_round_strokes_wind_reference`

## Estimated temperature impact
- `estimated_temperature_impact_strokes`

Definition:
- `predicted_round_strokes - predicted_round_strokes_temperature_reference`

## Estimated total weather impact
- `estimated_total_weather_impact_strokes`

Definition:
- `predicted_round_strokes - predicted_round_strokes_total_weather_reference`

These are descriptive counterfactual model outputs, not causal ground truth.

## DynamoDB Checkpoint Contract

## Checkpoint family
- `pk = PIPELINE#SCORE_ROUND_WIND_MODEL`
- `sk = EVENT#<event_id>#MODEL#<training_request_fingerprint>`

## Checkpoint purpose
This stores one logical scoring checkpoint per:
- event
- model artifact identity

This means the same event can be rescored against a different trained model without checkpoint collisions.

## Checkpoint fields
Typical fields include:
- `pipeline`
- `event_id`
- `training_request_fingerprint`
- `status`
- `last_run_id`
- `updated_at`
- `event_year`
- `rows_scored`
- `scored_rounds_key`
- `model_name`
- `model_version`
- `model_artifact_prefix`
- `scoring_request_fingerprint`

### Status values
Typical statuses:
- `success`
- `failed`

## Run summary
A run summary item is written with:
- `pk = RUN#<run_id>`
- `sk = SCORE_ROUND_WIND_MODEL#SUMMARY`

## Incremental and Idempotent Behavior

The scoring step is incremental at the event + model level.

## Scoring request fingerprint
A scoring request fingerprint is computed from:
- the scored source event object metadata
- the selected training request fingerprint
- the scoring policy version

## Skip behavior
An event is skipped when:
- a success checkpoint exists for the same event + training request fingerprint
- the stored scoring request fingerprint matches the current one
- `--force-events` is not used

## Idempotent writes
Each event writes to a deterministic output key:
- `gold/.../scored_rounds/event_year=<YYYY>/tourn_id=<ID>/scored_rounds.parquet`

Rerunning the same event/model scoring request overwrites the same logical location.

## Relationship to Upstream and Downstream Steps

## Upstream
- `gold_wind_model_inputs`
  - provides the canonical round model-input rows
- `train_round_wind_model`
  - provides the trained model artifact and feature contract

## Current step
- `score_round_wind_model`
  - produces row-level scored outputs and estimated weather impacts

## Downstream
- `report_round_weather_impacts`
  - builds dashboard-ready aggregate reporting tables from scored rows
- dashboard / notebook / visualization layer
  - consumes published reporting tables and scored round detail

## Notes on Scope

This step intentionally does not:
- train the model
- choose the latest model automatically
- aggregate dashboard summaries
- generate charts
- make causal claims

Its responsibility is:
- deterministic row-level scoring
- deterministic counterfactual scoring
- stable impact field production
- event-level incremental processing
